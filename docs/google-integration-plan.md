-# Google Calendar + Gmail Integration — Plan (for review)

Status: **proposal**. Nothing here is built yet. Canvas editor is being built in parallel.

## The MCP reality (read first)

The Gmail / Google Calendar **MCP connectors available inside the Claude Code session are authenticated to one Google account (the developer's) and live only in the assistant's tooling**. They cannot serve end-users of the deployed product — every student must act as themselves (send from *their* Gmail, write to *their* calendar).

So regardless of transport (an MCP toolset wired into the ADK agent, or direct Google client libraries), the product needs **per-user Google OAuth**: each user grants consent once, we store their refresh token, and the backend acts on their behalf. An MCP server does not remove this requirement — it would still need each user's credentials injected.

**Recommendation:** skip MCP for the product runtime. Use Google's official client libraries (`google-api-python-client`, `google-auth`) directly in the backend, with per-user OAuth tokens. (The session MCP tools remain useful to *me* for prototyping/testing.)

## Scopes

One consent screen, minimal scopes:

- `https://www.googleapis.com/auth/gmail.send` — send email as the user (send-only; cannot read inbox).
- `https://www.googleapis.com/auth/calendar.events` — create/update events on the user's calendar.

The app already uses Firebase Google sign-in, but Firebase Auth does **not** return Gmail/Calendar scopes or a storable refresh token. This is a **separate OAuth grant** (incremental authorization) on top of login.

## Data model

- New Firestore doc per user: `users/{uid}/integrations/google` → `{ refresh_token, scopes, connected_at, email }`. Refresh token is sensitive — store in this doc (Firestore rules already deny client access; backend-only via Admin SDK).
- New collection `users/{uid}/emails` (email drafts/sent log): `{ id, to, subject, body_markdown, kind: 'faculty'|'recommender', ref_id, status: 'draft'|'sent', linked_application_id?, sent_at?, created_at, updated_at }`.
- Tracker recommender entries gain an `email` field (collected in the edit modal; the `to` for LoR requests).
- Preferences gains `reminder_offsets_days: int[]` (default `[7, 1]`) — per-user reminder cadence.
- Applications store `calendar_event_id?` once a deadline event is created (for update/delete).

## Backend endpoints

OAuth:
- `GET /api/integrations/google/auth-url` → returns Google consent URL (state = uid).
- `GET /api/integrations/google/callback` → exchanges code, stores refresh token, redirects back to app.
- `GET /api/integrations/google/status` → `{ connected: bool, email }`.
- `DELETE /api/integrations/google` → revoke + delete stored token.

Email:
- `POST /api/emails` → create an email draft `{to, subject, body_markdown, kind, ref_id}`.
- `PATCH /api/emails/{id}` → edit (canvas-edited body).
- `POST /api/emails/{id}/send` → send via Gmail API as the user; on success set `status=sent`, and bump the linked recommender/faculty status (`asked` / `contacted`).
- `GET /api/emails` / `DELETE /api/emails/{id}`.

Calendar / reminders:
- `POST /api/calendar/deadline` `{application_id}` → create a Calendar event at the application deadline with **native reminder overrides** (e.g. email 7 days + 1 day before). Google then sends the reminder emails/notifications itself — **no transactional email infra needed**. Store the returned `event_id` on the application for later update/delete.
- Auto-hook: creating/updating a tracker deadline offers "add to Google Calendar".

## Agent (ADK) wiring

- New agent tool `draft_email(kind, recipient, context)` → produces `{subject, body_markdown}`, persisted as an email draft, surfaced through the **same HITL review → canvas-edit → approve** flow as drafts.
- On approve, the existing approval gate calls `POST /api/emails/{id}/send`.
- Reuses the result-card pattern: chat shows "Email ready · Open in …" instead of dumping the body.

## Outreach vs recommender emails

Unify under the `emails` collection with a `kind` discriminator:
- `kind='faculty'` — cold email to a professor (recipient from Shortlist `email`); on send → `outreach_status='contacted'`. The existing **outreach draft** becomes the email body source.
- `kind='recommender'` — LoR request to a referee (recipient from the new recommender `email`); on send → recommender `status='asked'`.

Both generated by the agent, edited in the canvas, sent via Gmail.

## Decisions (answered)

1. **OAuth client: reuse the existing Google Cloud project's client.** Add a backend redirect URI (e.g. `https://<backend-host>/api/integrations/google/callback`) to its Authorized redirect URIs. Enable the **Gmail API** + **Calendar API** in that project.
2. **Reminder cadence: configurable per user.** Store `reminder_offsets_days: int[]` in Preferences, default `[7, 1]`. Settings UI to edit; each tracker deadline event uses the user's offsets as Calendar reminder overrides.
3. **Recommender email: yes.** Add an `email` field per recommender — collected in the tracker edit modal (recommender rows get an email input), persisted on the recommender object, used as the `to` for LoR-request emails.
4. **App is `external` (public).** Implications baked into the plan below.

### External + `gmail.send` → verification burden (important)

`gmail.send` is a **restricted scope**. For an `external` app serving the public:
- Until the OAuth consent screen passes **Google verification** (incl. a likely **CASA security assessment** for restricted scopes), the app is capped at **100 test users** and shows an **"unverified app"** warning on consent.
- Verification needs: a published privacy policy URL, homepage, app domain verification, demo video, and the security assessment for the restricted Gmail scope. Can take **weeks**.

**Mitigation / sequencing impact:**
- **Calendar** uses `calendar.events` (a *sensitive*, not *restricted*, scope) — lighter review. Ship Calendar reminders first; it's usable for test users immediately and is the lower-verification path.
- For email, consider launching with `gmail.send` behind the 100-test-user cap while verification is in flight. If verification proves too heavy, fallback option is a "compose in Gmail" handoff (open a prefilled `mailto:`/Gmail compose URL with subject+body) — no restricted scope, user clicks send themselves. Flag this as the contingency.

## Sequencing (once approved)

1. OAuth connect flow + token storage + status UI (Settings → "Connected accounts").
2. Calendar deadline events + reminders (smallest win, no email-sending review needed).
3. Email model + Gmail send + agent `draft_email` tool + canvas review.
